# Reward Rule Entity

## Document Information

| Field              | Value                                           |
| ------------------ | ----------------------------------------------- |
| Project            | Credit Card Intelligence Platform (CCIP)        |
| Entity             | Reward Rule                                     |
| Primary Table      | `reward_rules`                                  |
| Supporting Tables  | `reward_rule_targets`, `reward_rule_exclusions` |
| Version            | 1.0                                             |
| Status             | Draft                                           |
| Architecture Scope | Recommendation Engine MVP                       |

---

## 1. Purpose

The `reward_rules` table stores the earning rules used by a credit card.

A reward rule defines:

* Which card the rule belongs to
* What type of reward is earned
* Where the rule applies
* How the reward is calculated
* Whether the reward is capped
* Whether minimum spending is required
* Which transactions are excluded
* When the rule is effective
* How the reward is converted into estimated SAR value

The Recommendation Engine uses reward rules to estimate the monetary value generated by each card for a user’s spending profile.

---

## 2. Core Calculation Flow

```text
User Spending Profile
↓
Spending Category or MCC
↓
Applicable Reward Rule
↓
Reward Quantity
↓
Reward Monetary Value
↓
Annual Estimated Reward Value
↓
Card Net Value
```

The reward engine must calculate the monetary value of rewards before cards can be compared.

---

## 3. Supported Reward Types

The MVP supports three primary reward types:

| Code       | Meaning                        |
| ---------- | ------------------------------ |
| `POINTS`   | Bank or loyalty-program points |
| `MILES`    | Airline or travel miles        |
| `CASHBACK` | Direct monetary cashback       |

Future reward types may include:

* Statement credit
* Merchant credit
* Vouchers
* Hotel points
* Tier points
* Promotional credits

---

## 4. Reward Rule Modeling Principle

Each independent earning condition must be stored as a separate reward rule.

Example:

A card earns:

```text
3 points per SAR on dining
2 points per SAR on groceries
1 point per SAR on general spending
```

This must create three separate reward-rule records.

The rules should not be stored in a single JSON field or unstructured text field.

---

## 5. Primary Table

Table name:

```text
reward_rules
```

Each record represents one reward-earning rule for one card.

## 7. Business Identifier

Each reward rule uses:

```text
id
code
```

Example:

```text
RJHI_ALFURSAN_INFINITY_DINING_3X
```

The `code` must be:

* Unique
* Stable
* Human-readable
* Independent from the database UUID

Recommended format:

```text
<BANK>_<CARD>_<CATEGORY>_<RATE>
```

---

## 8. Card Relationship

Each reward rule belongs to one card.

Foreign key:

```text
reward_rules.card_id
→ cards.id
```

Relationship:

```text
cards
1
│
└── many reward_rules
```

A card may have:

* One base reward rule
* Several category rules
* Several MCC-specific rules
* Several promotional rules
* Several exclusion rules

---

## 9. Loyalty Program Relationship

A reward rule may credit rewards into a loyalty program.

Foreign key:

```text
reward_rules.loyalty_program_id
→ loyalty_programs.id
```

Examples:

* AlFursan miles
* Bank reward points
* Cashback program
* Hotel loyalty points

The field is optional because cashback may not require a separate loyalty program.

## 10. Calculation Methods

Recommended MVP values:

| Value                   | Meaning                                    |
| ----------------------- | ------------------------------------------ |
| `PER_SPEND_UNIT`        | Fixed reward quantity per spend amount     |
| `PERCENTAGE`            | Cashback percentage of eligible spend      |
| `FIXED_PER_TRANSACTION` | Fixed reward for each eligible transaction |
| `TIERED`                | Reward depends on spending tier            |
| `NONE`                  | No reward                                  |

### PER_SPEND_UNIT

Example:

```text
3 points per 1 SAR
```

Stored as:

```yaml
calculation_method: PER_SPEND_UNIT
reward_rate: 3
spend_unit: 1
```

Another example:

```text
1 mile per 5 SAR
```

Stored as:

```yaml
calculation_method: PER_SPEND_UNIT
reward_rate: 1
spend_unit: 5
```

Formula:

```text
Reward Quantity
=
Eligible Spend
÷ Spend Unit
× Reward Rate
```

---

### PERCENTAGE

Example:

```text
5% cashback
```

Stored as:

```yaml
calculation_method: PERCENTAGE
reward_rate: 5
spend_unit: 100
```

Formula:

```text
Cashback Value
=
Eligible Spend
× Reward Rate
÷ 100
```

For percentage rules, the Recommendation Engine may directly treat the cashback result as monetary value.

---

### FIXED_PER_TRANSACTION

Example:

```text
10 SAR cashback per eligible transaction
```

Stored as:

```yaml
calculation_method: FIXED_PER_TRANSACTION
reward_rate: 10
spend_unit: 1
```

This method requires transaction-count information and may be used only when the user profile or calculator provides transaction frequency.

It is not a primary MVP calculation method.

---

### TIERED

Example:

```text
1 point per SAR for the first 5,000 SAR
2 points per SAR above 5,000 SAR
```

Tiered rules require supporting records.

Recommended future table:

```text
reward_rule_tiers
```

For the MVP, each tier may temporarily be represented as a separate rule with explicit thresholds.

---

## 13. Reward Target Types

Recommended values:

| Value               | Meaning                          |
| ------------------- | -------------------------------- |
| `GENERAL`           | Applies to all eligible spending |
| `SPENDING_CATEGORY` | Applies to one spending category |

---

## 14. Target Validation

The target table must enforce the following logic:

### General Rule

```text
target_type = GENERAL
spending_category_id = null
mcc_category_id = null
```

### Spending Category Rule

```text
target_type = SPENDING_CATEGORY
spending_category_id is required
mcc_category_id = null
```

### MCC Rule

```text
target_type = MCC
mcc_category_id is required
spending_category_id = null
```

A target record must not reference both a spending category and an MCC.

---

## 15. Base Rule

A base rule represents the default reward rate applied when no more specific rule matches.

Example:

```text
1 point per SAR on general eligible spending
```

Stored as:

```yaml
is_base_rule: true
target_type: GENERAL
priority: 100
```

Recommended constraint:

A card should have no more than one active base rule for the same reward program and effective period.

---

## 16. Rule Matching Priority

When multiple rules match the same spending, the engine should apply the most specific valid rule.

Recommended priority order:

```text
1. Exact MCC rule
2. MCC group rule
3. Spending category rule
4. General base rule
5. No reward
```

The `priority` field resolves conflicts between rules with the same specificity.

Recommended interpretation:

```text
Lower number = higher priority
```

Example:

```text
priority 10
```

takes precedence over:

```text
priority 50
```

Specificity should be evaluated before numerical priority.

---

## 17. Reward Rule Exclusions

Reward exclusions should not be embedded as free-text notes.

A separate table should be used.

Table name:

```text
reward_rule_exclusions
```

An exclusion defines transactions that are not eligible under a specific rule.

Recommended MVP values:

| Value               | Meaning                              |
| ------------------- | ------------------------------------ |
| `SPENDING_CATEGORY` | Excludes an entire spending category |
| `MCC`               | Excludes one MCC                     |
| `CASH_LIKE`         | Excludes cash-equivalent activity    |

## 20. Caps

Reward rules may have limits.

Two cap types are supported:

### Reward Quantity Cap

Example:

```text
Maximum 2,000 points per month
```

Stored as:

```yaml
maximum_reward_amount: 2000
cap_period: MONTHLY
```

### Eligible Spend Cap

Example:

```text
5% cashback on the first 1,000 SAR per month
```

Stored as:

```yaml
maximum_eligible_spend: 1000
cap_period: MONTHLY
```

When both are provided, the most restrictive cap must be applied.

---

## 21. Cap Period Values

Recommended values:

| Value             | Meaning              |
| ----------------- | -------------------- |
| `TRANSACTION`     | Per transaction      |
| `DAILY`           | Per calendar day     |
| `MONTHLY`         | Per calendar month   |
| `QUARTERLY`       | Per calendar quarter |
| `ANNUAL`          | Per calendar year    |
| `STATEMENT_CYCLE` | Per billing cycle    |
| `CAMPAIGN`        | During a promotion   |
| `LIFETIME`        | Lifetime of the rule |

For recommendation estimates, the engine may normalize monthly and annual caps into annual values.

## 22. Minimum Spending

* Minimum transaction amount
* Minimum monthly spend
* Minimum statement-cycle spend
* Minimum annual spend

```text
minimum_transaction_amount

```

```text
minimum_spend_period
```

| Field                  | Type | Required | Notes                                  |
| ---------------------- | ---- | -------: | -------------------------------------- |
| `minimum_spend_period` | Text |       No | Period used by minimum-period spending |

## 23. Rounding Methods

Reward programs may round rewards differently.

Recommended values:

| Value                 | Meaning                                |
| --------------------- | -------------------------------------- |
| `NONE`                | No rounding                            |
| `FLOOR_TRANSACTION`   | Round down for each transaction        |
| `FLOOR_TOTAL`         | Round down after period aggregation    |
| `NEAREST_TRANSACTION` | Round each transaction to nearest unit |
| `NEAREST_TOTAL`       | Round period total to nearest unit     |

Default:

```text
FLOOR_TOTAL
```

The exact method should come from official program terms when available.

---

## 24. Reward Valuation

Points and miles cannot be compared directly.

The platform must convert them into estimated SAR value.

Approved architecture decision:

```text
Fixed Reference Value
```

Each loyalty program should have a reference redemption value.

Example:

```text
1 AlFursan mile = 0.05 SAR
1 bank point = 0.01 SAR
1 cashback SAR = 1 SAR
```

Formula:

```text
Reward Monetary Value
=
Reward Quantity
× Reference Unit Value
```

The reward rule itself stores the earning rate.

The loyalty-program entity stores the reference monetary value.

---

## 25. Cashback Valuation

Cashback has direct monetary value.

Example:

```text
100 SAR cashback
=
100 SAR reward value
```

For cashback:

```text
Reference Value = 1 SAR per cashback SAR
```

No external conversion is required.

---

## 26. Estimated Annual Reward Value

For every spending category:

```text
Annual Eligible Spend
=
Monthly Spend × 12
```

Then:

```text
Reward Quantity
=
Applicable Reward Formula
```

Then:

```text
Estimated Reward Value
=
Reward Quantity × Reference Value
```

Total card reward value:

```text
Total Annual Reward Value
=
Sum of all category reward values
```

Future net-value calculation:

```text
Net Value
=
Rewards
+ Offers
+ Benefits
− Annual Fees
```

---

## 27. Overlapping Categories

A user’s spending must not be counted twice.

Example:

```text
TRAVEL
├── AIRLINES
└── HOTELS
```

If the user enters spending separately for airlines and hotels, the engine must not also add the same amount under general travel.

Recommended rule:

* Use the most detailed user-entered category.
* Parent categories may be used only when child values are not supplied.
* Spending-category hierarchy must be respected during aggregation.

---

## 28. Promotional Rules

Temporary earning campaigns use:

```text
is_promotional = true
effective_from
effective_to
```

Example:

```text
10% cashback on dining from 1 July to 31 August
```

Promotional rules must not remain active after `effective_to`.

For MVP recommendations, the engine may:

* Exclude promotional rules from long-term annual rankings
* Display them separately
* Include them only when explicitly calculating a promotional period

Default MVP behavior:

```text
Do not include temporary promotional rules in standard annual recommendation ranking.
```

---

## 29. Current-State Strategy

The project uses a current-state data model.

Historical reward-rule versions are not required for MVP.

When a rule changes:

1. Update the current rule
2. Update `effective_from`
3. Update `last_verified_at`
4. Preserve the official source reference

Future versions may create immutable rule versions.

---

## 30. Effective Dates

Rules are active only when:

```text
is_active = true
AND effective_from <= calculation_date
AND (
    effective_to is null
    OR effective_to >= calculation_date
)
```

If `effective_from` is unknown, the rule may remain active only after manual verification.

---

## 31. Source Traceability

Every production reward rule should include:

```text
source_url
source_reference
last_verified_at
```

Examples of `source_reference`:

* Page 4
* Rewards section
* Clause 7.2
* Table titled “Earning Rates”
* Product terms dated January 2026

Accepted sources:

* Official bank product pages
* Official terms and conditions
* Official loyalty-program documentation
* Official tariff guides
* Official promotional terms

Editorial summaries must not be treated as authoritative sources.

---

## 32. Constraints

### Reward Rules

Required constraints:

* `id` must be unique.
* `code` must be unique.
* `card_id` must reference an existing card.
* `loyalty_program_id`, when provided, must reference an existing loyalty program.
* `currency_id` must reference an existing currency.
* `reward_rate` must be zero or greater.
* `spend_unit` must be greater than zero.
* `minimum_transaction_amount` must be zero or greater.
* `minimum_period_spend` must be zero or greater.
* `maximum_reward_amount` must be greater than zero when provided.
* `maximum_eligible_spend` must be greater than zero when provided.
* `priority` must be zero or greater.
* `effective_to` must not be earlier than `effective_from`.
* `last_verified_at` must not be earlier than `effective_from`.
* A rule with `reward_type = CASHBACK` should normally use `calculation_method = PERCENTAGE` or `FIXED_PER_TRANSACTION`.
* A rule must have at least one target.
* An active base rule must have a `GENERAL` target.

### Reward Rule Targets

* `reward_rule_id` must reference an existing reward rule.
* A target must reference only one target object.
* Duplicate targets for the same rule are not allowed.

### Reward Rule Exclusions

* `reward_rule_id` must reference an existing reward rule.
* An exclusion must contain enough information to identify what is excluded.
* Duplicate exclusions for the same rule are not allowed.

## 38. Recommendation Engine Matching Process

For each card and spending input:

```text
1. Confirm card is recommendation eligible
2. Load active reward rules
3. Remove expired rules
4. Identify matching MCC or spending category
5. Apply exclusions
6. Select the most specific rule
7. Resolve rule priority
8. Apply minimum-spend requirements
9. Apply eligible-spend cap
10. Calculate reward quantity
11. Apply rounding
12. Apply reward cap
13. Convert reward into SAR
14. Aggregate annual card reward value
```

## 40. Architecture Summary

```text
cards
1
│
└── many reward_rules
          │
          ├── many reward_rule_targets
          │       ├── spending_categories
          │       └── mcc_categories
          │
          └── many reward_rule_exclusions
                  ├── spending_categories
                  └── mcc_categories
```

This design allows the Recommendation Engine to calculate comparable annual reward value while keeping earning rules structured, traceable, and extensible.
