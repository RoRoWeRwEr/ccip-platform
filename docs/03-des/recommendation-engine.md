---

# 1. Purpose

The Recommendation Engine is the core intelligence component of the Credit Card Intelligence Platform (CCIP).

Its purpose is to analyze a user's spending profile, compare it against all eligible credit cards, estimate the annual value generated by each card, and rank cards according to their expected net value.

The engine must produce transparent, explainable, and reproducible recommendations.

---

# 2. Objectives

The engine shall:

* Estimate annual reward value.
* Estimate annual cashback value.
* Estimate annual points value.
* Estimate annual airline miles value.
* Apply spending limits and caps.
* Apply exclusions.
* Apply minimum spend requirements.
* Convert all rewards into estimated SAR.
* Subtract annual fees.
* Rank cards.
* Explain why every recommendation was produced.

---

# 4. Recommendation Scope

Only cards satisfying all conditions are evaluated.

```text
is_active = true
AND
is_public = true
AND
is_recommendation_eligible = true
AND
availability_status = AVAILABLE
```

Cards failing any condition are ignored.

---

# 5. Processing Pipeline

The Recommendation Engine executes the following stages.

```text
User Inputs
↓
Validate Inputs
↓
Normalize Spending
↓
Load Eligible Cards
↓
Load Reward Rules
↓
Match Spending Categories
↓
Apply Rule Priority
↓
Apply Exclusions
↓
Apply Caps
↓
Calculate Reward Quantity
↓
Convert Rewards to SAR
↓
Subtract Annual Fees
↓
Generate Net Value
↓
Rank Cards
↓
Generate Explanation
↓
Return Results
```

---

# 6. Spending Normalization

Monthly values become annual values.

Formula:

```text
Annual Spend = Monthly Spend × 12
```

The engine works internally with annual spending.

---

# 7. Reward Rule Matching

For every spending category:

1. Load applicable reward rules.
2. Ignore inactive rules.
3. Ignore expired rules.
4. Apply exclusions.
5. Select the most specific rule.

Priority:

```text
Exact MCC
↓
MCC Group
↓
Spending Category
↓
General Rule
```

---

# 8. Reward Calculation

For every matched rule:

## Points

```text
Reward Quantity

=

Spend

÷ Spend Unit

×

Reward Rate
```

---

## Cashback

```text
Cashback

=

Eligible Spend

×

Percentage
```

---

## Miles

```text
Miles

=

Spend

÷ Spend Unit

×

Reward Rate
```

---

# 9. Caps

The engine applies caps after reward calculation.

Supported caps:

* Monthly
* Annual
* Statement Cycle
* Campaign

The most restrictive cap always wins.

---

# 10. Exclusions

Before calculating rewards:

The engine removes spending excluded by:

* MCC
* Spending Category
* Cash-like Transactions
* Government Transactions
* Fees
* Refunds

Excluded spending contributes zero reward.

---

# 11. Minimum Spend

If a rule requires minimum spending:

```text
Eligible?

YES
↓

Reward

NO
↓

Zero
```

---

# 12. Reward Valuation

All rewards become SAR.

Example:

```text
AlFursan

```

```text
Reward Value

=

Reward Quantity

×

Reference Value
```

Cashback:

```text
1 SAR Cashback

1 SAR
```

# 13. Annual Reward Value

For every spending category:

```text
Reward Value

↓

Category Total
```

All categories:

```text
Dining

+

Fuel

+

Travel

+

Groceries

+

General

=

```

---

# 14. Net Value

Approved project formula:

```text
Net Value

=

Rewards

+

Offers

+

Benefits

−

Annual Fees
```

For MVP:

Offers and Benefits are not yet implemented.

Therefore:

```text
Net Value

=

Reward Value

−

Annual Fee
```

The architecture already supports future expansion without changing the formula.

---

# 15. Existing Cards

If the user owns a card:

The engine calculates:

```text
Current Card Value
```

Then:

```text
Candidate Card Value
```

Finally:

```text
Incremental Value

=

Candidate

−

Current
```

Future recommendations may include:

* Replace current card
* Keep current card
* Add complementary card

---

# 16. Ranking

Cards are sorted by:

```text
Highest Net Value
```

Tie breaker order:

1. Higher Reward Value
2. Lower Annual Fee
3. Higher Recommendation Score
4. Alphabetical Name

---

# 17. Recommendation Score

Besides Net Value, every card receives an internal score.

Range:

```text
0–100
```

Example weighting:

| Factor         | Weight |
| -------------- | -----: |
| Net Value      |    60% |
| Goal Alignment |    20% |
| Spending Match |    10% |
| Fee Efficiency |    10% |

The Recommendation Score is primarily for ranking and future tuning.

---

# 18. Goal Alignment

Different goals affect presentation, not reward calculation.

Examples:

Cashback users:

* Cashback cards first.

Miles users:

* Airline cards first.

General Value:

Pure Net Value ranking.

Future versions may allow multiple weighted goals.

---

# 19. Recommendation Explanation

Every recommendation must explain itself.

Example:

```text
Why Recommended

• Highest dining rewards
• Excellent travel earning
• Annual fee recovered after 3 months
• Strong match for your spending profile
```

Users should never receive an unexplained ranking.

---

# 20. Confidence Level

Every recommendation receives:

```text
Confidence

HIGH

MEDIUM

LOW
```

Confidence depends on:

* Completeness of user inputs.
* Rule verification status.
* Missing assumptions.

---

# 21. Missing Data

If spending data is incomplete:

Example:

```text
Travel

Unknown
```

The engine:

* Assumes zero spending.
* Reduces confidence.

It must not invent spending.

---

# 22. Annual Fee Recovery

Useful explanation:

```text
Annual Fee

```

```text
1,800 SAR
```

Recovery:

```text
500

÷

1,800

=

3.3 months
```

This value is informational.

# 24. Returned Fields

Minimum API output:

* Card ID
* Card Name
* Annual Reward Value
* Annual Fee

* Recommendation Reasons

---

# 25. Auditability

Every recommendation should be reproducible.

Given identical:

* Card data
* Reward rules
* Spending profile
* Reference values

The engine must always produce identical results.

No randomness is allowed.

---

# 26. Performance Goals

Target response:

```text
< 2 seconds
```

for a complete recommendation request.

The engine should scale to hundreds of cards without architectural changes.

---

# 27. MVP Limitations

Deferred:

* Welcome bonuses
* Promotional campaigns
* Merchant-specific offers
* Day-of-week promotions
* Time-of-day rewards
* AI personalization
* Customer segmentation
* Dynamic reward valuation
* Portfolio optimization
* Multi-card spending optimization

These will be implemented in later releases.

---

# 28. Future Enhancements

Planned capabilities:

* Spending simulation

* Redemption optimization

# 29. Design Principles

The Recommendation Engine must always be:

* Transparent
* Deterministic
* Explainable
* Traceable
* Extensible
* Data-driven
* Independent of UI
